Data Storage & External Services

The application uses a single Oracle Database Free 23ai instance as its only data store. Photo binary data is persisted directly as BLOB columns in the PHOTOS table alongside metadata such as filename, MIME type, dimensions, file size, and upload timestamp. There are no external caches, message brokers, or third-party API integrations; all data flows through JDBC/Hibernate to Oracle. In the Docker Compose deployment, Oracle runs as a sidecar container (photoalbum-oracle) on the same private bridge network as the application container.

Key Architectural Decisions

  • Database-centric BLOB storage: Photo binary data is stored directly in Oracle as BLOB columns rather than on a filesystem, simplifying deployment at the cost of larger DB footprint.
  • Oracle-specific native queries: The PhotoRepository uses Oracle-specific SQL constructs (ROWNUM, NVL, TO_CHAR, analytic RANK()/SUM() OVER) for pagination, filtering, and statistics.
  • Constructor-injected Spring services: All controllers and services use constructor injection for dependency management, with @Transactional scoping defined at the service implementation level.

Primary Workflows

Workflow 1: Upload Photos

A user selects one or more image files on the gallery page and submits them via POST /upload. Each file is processed independently by PhotoService.uploadPhoto:

  1. File type validation — MIME type is checked against the allowed list (JPEG, PNG, GIF, WebP); rejected files return an error entry.
  2. File size validation — size must be > 0 and ≤ 10 MB (configurable via app.file-upload.max-file-size-bytes); oversized files return an error entry.
  3. Image dimension extraction — the byte array is decoded with ImageIO to extract width and height; failure is non-fatal (dimensions remain null).
  4. UUID generation — a random UUID is assigned as the photo ID and as the stored filename.
  5. Persistence — a Photo entity is created with the full byte array as photoData and saved to Oracle via PhotoRepository.save() inside a @Transactional method.
  6. Response assembly — successfully uploaded photos and failed uploads are returned as separate JSON lists; the overall success flag is true only when at least one upload succeeded.

Workflow 2: Browse Photo Gallery

A user navigates to GET /. HomeController.index calls PhotoService.getAllPhotos(), which queries the database for all photos ordered by uploadedAt descending (newest first). The photo list is passed to the index Thymeleaf template. A timestamp is added to the model for client-side cache-busting of thumbnails.

Workflow 3: View Photo Detail with Navigation

A user clicks a photo thumbnail, triggering GET /detail/{id}. DetailController.detail validates the ID, retrieves the Photo entity, and then resolves sequential navigation by querying for the nearest photo uploaded before (previous) and the nearest photo uploaded after (next). The detail template receives the photo metadata plus optional previous/next photo IDs for navigation links.

Workflow 4: Serve Photo Binary Data

When a browser requests a photo image at GET /photo/{id}, PhotoFileController.servePhoto fetches the Photo entity, reads the photoData BLOB byte array, and streams it back as a ByteArrayResource with the stored MIME type. Response headers are set to prevent caching (Cache-Control: no-cache, no-store, must-revalidate).

Workflow 5: Delete Photo

A user submits POST /detail/{id}/delete on the detail page. DetailController.deletePhoto calls PhotoService.deletePhoto, which confirms the photo exists and then removes it from the database. On success or failure, a flash message is added to redirect attributes before the user is redirected to the gallery.

Cross-Service Data Flows

PhotoAlbum is a single-service monolith; all business logic resides in one Spring Boot application backed by one Oracle database. There are no cross-service network calls or API gateway aggregation patterns.

Data flows within the application tier as follows:

  • Upload path: Browser → HomeController → PhotoServiceImpl (validates, extracts dimensions, builds entity) → PhotoRepository → Oracle DB (stores BLOB + metadata)
  • Gallery path: Browser → HomeController → PhotoServiceImpl → PhotoRepository → Oracle DB (returns metadata rows) → Thymeleaf template rendered to browser
  • Binary serving path: Browser → PhotoFileController → PhotoServiceImpl → PhotoRepository → Oracle DB (returns BLOB bytes) → streamed as HTTP response body
  • Detail/navigation path: Browser → DetailController → PhotoServiceImpl (three separate queries: one for the current photo, one for the previous, one for the next) → Oracle DB → Thymeleaf template

Business Rules & Decision Logic

Validation Rules

  • MIME type check: Only image/jpeg, image/png, image/gif, and image/webp are accepted. Any other content type rejects the file with the message "File type not supported. Please upload JPEG, PNG, GIF, or WebP images."
  • File size upper bound: Files larger than app.file-upload.max-file-size-bytes (default 10 MB) are rejected with a message stating the MB limit.
  • Empty file check: Files with a size of 0 bytes are rejected with the message "File is empty."
  • ID presence check: Requests to /photo/{id} or /detail/{id} with a null/blank ID are short-circuited (404 or redirect to gallery).

Decision Logic

  • Partial batch success: In a multi-file upload, each file is processed independently. The overall success flag in the JSON response is true only when at least one file was uploaded successfully. All individual failures are still reported alongside successes.
  • Navigation boundaries: If no photo exists before or after the current photo's upload timestamp, the previous or next navigation link is omitted (the template receives null for the corresponding ID).
  • Image dimension extraction: If ImageIO cannot decode the image (e.g., corrupt data or unsupported internal format), the upload still succeeds but width and height are stored as null — non-fatal degradation.

State Transitions

  • Photo lifecycle: Uploaded → Persisted (after successful DB save) → Deleted (after DELETE). No intermediate states or soft-delete mechanism; deletion is permanent.

Transactions

  • getAllPhotos, getPhotoById, getPreviousPhoto, and getNextPhoto run in read-only transactions.
  • uploadPhoto and deletePhoto run in the default (read-write) transaction inherited from the @Transactional class-level annotation on PhotoServiceImpl.

Error Handling

  • Runtime exceptions from database operations are caught and re-thrown as RuntimeException, propagating up to controller-level try/catch blocks which log the error and return either a 500 response or redirect with a flash error message.
  • Upload failures at the file-validation level are captured in UploadResult objects and returned to the client as structured JSON — they do not throw exceptions.

Authorization

No authentication or authorization mechanism is implemented; all gallery, upload, detail, and delete operations are accessible to any user.

Version & Compatibility Risks

Spring Boot 2.7.18 reached end-of-life in November 2023 and no longer receives security patches; migration to Spring Boot 3.x (which requires Java 17+ and Jakarta EE 10 namespace changes from javax. to jakarta.) is the recommended upgrade path. The project targets Java 8, which is several LTS releases behind — Java 17 is the minimum for Spring Boot 3.x and Java 21 is the current LTS. Oracle JDBC ojdbc8 version is managed by the Spring Boot BOM (typically resolves to 21.x), but the Oracle driver introduces a proprietary runtime dependency that can complicate containerized or cloud-native deployments. commons-io 2.11.0 is not the latest release (2.16.x as of 2024) but carries no known critical CVEs.

Notable Observations

  • Vendor lock-in via Oracle JDBC: The sole database driver is ojdbc8, tightly coupling the application to Oracle Database. Migrating to a cloud-managed database (e.g., PostgreSQL on Azure) would require a driver swap and potential schema changes.
  • Spring Boot 2.7.x EOL: The parent BOM is end-of-life; upgrading to Spring Boot 3.x requires a Java version bump to 17+ and namespace migration from javax. to jakarta. throughout the codebase.
  • No caching, messaging, or security libraries declared: The application has no declared caching, message broker, or authentication/authorization dependencies, indicating a minimal, stateless design without dedicated security controls.
  • DevTools included as optional: spring-boot-devtools is marked optional and is excluded from production JARs by the Maven plugin, which is correct practice — however it signals the project may not have a clear separation between dev and prod build profiles.

Caching Strategy

No caching layer is configured. There are no Spring Cache annotations (@Cacheable, @CacheEvict), no EhCache, Redis, or Caffeine dependencies, and no Hibernate second-level cache configuration. All reads hit Oracle directly on every request.

Rationale note: Given that photo binary data (BLOB) is retrieved on every gallery load, adding a caching layer (e.g., Caffeine for metadata, or a CDN/blob cache for image bytes) would be a significant performance improvement opportunity.

Data Ownership Boundaries

The application is a single-service monolith. There is only one data store (Oracle Database), one schema (photoalbum user/schema), and one table (PHOTOS). There are no cross-service data access patterns, no read/write splitting, and no CQRS separation.

Data store topology: Shared single-instance Oracle database with the application owning its schema exclusively. In the Docker Compose environment, the Oracle XE container and the Spring Boot container are co-located on the same Docker network with the JDBC URL resolving via the oracle-db hostname.

Binary data storage: Photo binary content is stored as a BLOB column directly in Oracle rather than on a file system or object store. This means the database is both the metadata store and the binary asset store, which impacts backup size, query performance, and portability.
