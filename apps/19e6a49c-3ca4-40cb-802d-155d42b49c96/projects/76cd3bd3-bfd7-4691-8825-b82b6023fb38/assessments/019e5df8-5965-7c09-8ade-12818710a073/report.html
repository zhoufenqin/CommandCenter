Data Storage & External Services

The application uses a single Oracle Database Free 23ai instance (schema FREEPDB1) as its sole data store. All photo data — including the raw image bytes — is stored as a BLOB column (PHOTO_DATA) in the PHOTOS table alongside metadata such as filename, MIME type, file size, dimensions, and upload timestamp. There are no external file system uploads, no caching layer (e.g., Redis), no message broker, and no third-party API integrations. In the Docker Compose environment, a named volume (oracle_data) provides persistence for the Oracle data files.

Key Architectural Decisions

  • BLOB-in-database storage: Photo binary data is stored directly as Oracle BLOB columns rather than on the file system, simplifying deployment at the cost of database size growth.
  • Oracle-specific queries: The PhotoRepository uses native SQL with Oracle-specific constructs (ROWNUM, TO_CHAR, NVL, analytical RANK() OVER) instead of portable JPQL, tightly coupling persistence to Oracle.
  • Single-service monolith: The entire application is packaged as a single Spring Boot JAR with all concerns (web, business logic, data access) co-located, deployed alongside Oracle DB via Docker Compose.

Primary Workflows

Workflow 1: Upload Photos

A user selects one or more image files and submits them via POST /upload. The controller iterates over each file and delegates to PhotoService.uploadPhoto().

Steps:

  1. Receive upload request — HomeController receives List<MultipartFile> via POST /upload; rejects immediately if the file list is null or empty.
  2. Validate MIME type — PhotoServiceImpl checks file.getContentType() against the configured allowed list (image/jpeg, image/png, image/gif, image/webp). Unsupported types are rejected with an error message.
  3. Validate file size — File size must be greater than 0 and must not exceed app.file-upload.max-file-size-bytes (default 10 MB). Violations are rejected with a human-readable message.
  4. Read binary data — File bytes are read into memory (file.getBytes()).
  5. Extract image dimensions — ImageIO.read() is used to decode the image and obtain width and height. Failure to extract dimensions is non-fatal; the photo is still saved without dimension metadata.
  6. Persist to database — A Photo entity is constructed with a UUID-based ID, binary BLOB data, stored filename, compatibility path, size, and MIME type. PhotoRepository.save() persists the record (including BLOB) to Oracle.
  7. Build response — For each successfully saved photo the controller fetches the saved record and adds it to the uploadedPhotos list; failures go to failedUploads. A JSON response is returned.

Workflow 2: Browse Gallery

A user navigates to the home page to see all uploaded photos in reverse-chronological order.

Steps:

  1. Receive gallery request — HomeController handles GET /.
  2. Retrieve all photos — PhotoServiceImpl.getAllPhotos() calls PhotoRepository.findAllOrderByUploadedAtDesc(), which executes a native Oracle query ordering by UPLOADED_AT DESC.
  3. Render gallery view — The photo list (including metadata but excluding raw BLOB data) is added to the Thymeleaf model and the index template is rendered. On error, an empty list is shown and the error is logged.

Workflow 3: View Photo Detail with Navigation

A user clicks a photo thumbnail to view it full-size, with previous/next navigation controls.

Steps:

  1. Receive detail request — DetailController handles GET /detail/{id}. Null or blank IDs redirect to /.
  2. Load photo record — PhotoService.getPhotoById(id) fetches the photo entity. If not found, the user is redirected to /.
  3. Resolve navigation — PhotoService.getPreviousPhoto() queries findPhotosUploadedBefore() (timestamp-based, Oracle ROWNUM-limited) and PhotoService.getNextPhoto() queries findPhotosUploadedAfter(). The IDs of the adjacent photos are passed to the template.
  4. Render detail view — The detail Thymeleaf template is rendered with photo metadata and navigation IDs.
  5. Serve binary image — The browser requests the actual image bytes from PhotoFileController via GET /photo/{id}. The controller retrieves the BLOB from the Photo entity and streams it as a ByteArrayResource with the correct Content-Type and no-cache headers.

Workflow 4: Delete Photo

A user deletes a photo from the detail view via POST /detail/{id}/delete.

Steps:

  1. Receive delete request — DetailController handles POST /detail/{id}/delete.
  2. Locate photo — PhotoServiceImpl.deletePhoto(id) calls PhotoRepository.findById(id). Returns false (not found) if the ID is unknown.
  3. Remove record — PhotoRepository.delete(photo) removes the entity and its BLOB data from Oracle within the active transaction.
  4. Redirect with feedback — On success, a flash attribute successMessage is set; on failure, errorMessage. The user is redirected to the gallery at /.

Cross-Service Data Flows

This application is a single-service monolith with no inter-service communication. All data flows are contained within the single JVM process:

  • Browser → Controller → Service → Repository → Oracle DB: Every request follows this linear path.
  • BLOB retrieval: When the browser loads a photo page, a secondary HTTP request is issued to GET /photo/{id}. The PhotoFileController fetches the full Photo entity (including BLOB bytes) and returns the binary data directly in the HTTP response body, avoiding any filesystem intermediary.
  • No external service dependencies: There are no message queues, external APIs, or caching layers. All state is owned by the Oracle database.

Business Rules & Decision Logic

Validation Rules

Non-null file listHomeControllerEmpty or null file parameter returns HTTP 400

Decision Logic

  • Upload result aggregation: The upload endpoint processes all files in the batch regardless of individual failures. A mixed result (some succeeded, some failed) is valid; response.success is true as long as at least one file succeeded.
  • Dimension extraction is best-effort: If ImageIO.read() fails to decode the image, width and height are stored as null. The photo is still saved — missing dimensions are non-fatal.
  • Navigation boundary handling: When a photo has no predecessor (oldest) or no successor (newest), the corresponding navigation ID is null and the template hides the navigation control.

State Transitions

Photos have a simple two-state lifecycle:

  • Uploaded → record exists in Oracle DB with BLOB data
  • Deleted → record removed from Oracle DB (no soft-delete; deletion is permanent)

Transactions

  • PhotoServiceImpl is annotated @Transactional at the class level; all write operations (upload, delete) execute within a single transaction boundary.
  • Read-only operations (getAllPhotos, getPhotoById, getPreviousPhoto, getNextPhoto) use @Transactional(readOnly = true) for optimized database access.

Error Handling

  • Upload failures (validation or DB error) are captured per-file and returned as structured JSON rather than propagating as HTTP 500 errors, so partial batch uploads succeed gracefully.
  • Detail and file-serving controllers catch all exceptions, log them, and return a redirect or HTTP 500 rather than exposing stack traces.

Authorization

No authentication or authorization is implemented. All endpoints are publicly accessible.

Version & Compatibility Risks

Spring Boot 2.7.18 is the final release of the 2.7.x line, which reached end-of-OSS-support in November 2023. Upgrading to Spring Boot 3.x requires a baseline of Java 17, but this project targets Java 8 (maven.compiler.source=8), making an upgrade non-trivial. Spring Boot 3.x also migrates from javax. to jakarta. namespaces, which affects all JPA, Bean Validation, and Servlet API usages. The Oracle JDBC driver ojdbc8 is tied to Oracle Database and would need replacing or re-evaluation if the database is migrated to a different engine. commons-io 2.11.0 is behind the current 2.15.x release, though no critical CVEs are known for 2.11.0 at this time.

Notable Observations

  • Java 8 target is a blocking concern: Spring Boot 3.x and Spring Framework 6.x require Java 17 as a minimum; the project must upgrade the JDK before adopting modern Spring Boot versions.
  • Oracle vendor lock-in: The ojdbc8 driver and likely Oracle-specific SQL/dialect usage in the application create a hard dependency on Oracle Database, complicating cloud-native or multi-DB migrations.
  • No security or caching layer declared: The project has no Spring Security, OAuth, or caching dependency (e.g., Redis, EhCache), which may be a gap for production-readiness.
  • Thin dependency surface is a positive sign: With only 8 runtime dependencies all managed by a single parent BOM, the upgrade surface is small and well-controlled, making modernization tractable.

Caching Strategy

No caching layer is configured in this application.

  • No Spring Cache annotations (@Cacheable, @CacheEvict, @CachePut) are present in any service or repository class.
  • No second-level Hibernate cache provider (EhCache, Caffeine, Redis) is declared in pom.xml or any properties file.
  • No JSR-107 / JCache dependency is included.
  • Photo binary data (BLOB) is fetched directly from Oracle on every request, including gallery listing and individual detail views.

Implication: Each page load for the gallery executes a full table scan ordered by uploaded_at with no result caching. As the PHOTOS table grows, this will increasingly impact response times. The idx_photos_uploaded_at index on uploaded_at partially mitigates sort cost but does not offset repeated full BLOB column fetches.

Data Ownership Boundaries

The application is a single-module monolith with one deployable service. There is no multi-service topology and therefore no cross-service data access to document.

Data store topology: Single shared Oracle instance; all application data resides in one schema (photoalbum) in one database. The oracle-init/ scripts create this schema user, and Hibernate owns the PHOTOS table DDL.

Read/write patterns:

  • All writes (INSERT on upload, DELETE on remove) go through PhotoRepository.save() / PhotoRepository.delete() within a @Transactional boundary.
  • All reads use @Transactional(readOnly = true) service methods delegating to native SQL repository queries.
  • No CQRS or event sourcing patterns are in use; reads and writes share the same Oracle data source.
