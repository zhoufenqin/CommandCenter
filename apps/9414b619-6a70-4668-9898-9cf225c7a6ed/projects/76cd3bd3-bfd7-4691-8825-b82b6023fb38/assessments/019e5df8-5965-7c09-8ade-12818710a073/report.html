Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) accessed via raw ADO.NET. It queries four tables: Customers, Accounts, AccountTypes, and Transactions. No ORM or query abstraction layer is used — all SQL is inline in code-behind files. Two external HTTP services are also integrated: the ZavaAuthGateway handles employee authentication through Forms Authentication redirects (login/logout URLs are configured in web.config), and the Zava Ledger Service (zava-ledger:8080) is called via HttpWebRequest to retrieve real-time account balances and transaction history as XML. The application falls back to querying the local Transactions SQL table if the Ledger Service is unavailable.

Key Architectural Decisions

  • No service or business logic layer: all data access, external HTTP calls, and business logic are implemented directly inside ASP.NET Web Forms code-behind classes, following a single-tier code-behind pattern.
  • Delegated authentication via ZavaAuthGateway: the application does not handle credential validation itself; unauthenticated requests are redirected to an external auth gateway, which sets the Forms Authentication cookie upon success.
  • Dual data sources for transactions: transaction data is fetched primarily from the Ledger Service (HTTP/XML), with a local SQL Server fallback, providing resilience but introducing potential data consistency concerns.

DTOs & Contracts

ZavaAccountManager does not define formal DTO classes. Data flows through the following patterns:

  • DataTable (System.Data): Used as the in-process data model for customer lists, account lists, account types, and transaction history. DataTables are populated from SqlDataAdapter fills and bound directly to GridView controls. No serialization or immutability guarantees.
  • XmlDocument (System.Xml): Used as the deserialization model for responses from the ZavaLedger service. Nodes are accessed via XPath selectors (//balance, //availableBalance, //transaction). No typed contract classes exist — the XML schema is implicitly assumed.
  • No OpenAPI/Swagger spec, protobuf schemas, or GraphQL schemas were found.
  • No request/response DTO classes (records, POJOs, or C# model classes) exist in this project.

Refer to data-architecture.md for field-level details on the Customers, Accounts, AccountTypes, and Transactions database entities.

Communication Patterns

Synchronous HTTP (Outbound)

  • Default.aspx calls ZavaLedger via System.Net.HttpWebRequest (blocking synchronous I/O) for two operations: GET /api/accounts/{id}/balance and GET /api/accounts/{id}/transactions. Responses are parsed as XML using XmlDocument.Load(stream).
  • Base URL is read from AppSettings["LedgerBaseUrl"] (default: http://zava-ledger:8080).
  • No retry logic, circuit breaker, or timeout configuration is applied to these calls. Any network failure or non-200 response is caught by a bare catch {} block, which silences the error.

Fallback Behavior

  • If the ZavaLedger transactions call fails, the application falls back to a direct SQL query against the local ZavaBankDB database (SELECT TOP 25 ... FROM Transactions WHERE AccountID=@a). No fallback is implemented for the balance call — it returns the string "Ledger balance unavailable." on failure.

Authentication Redirect (Outbound)

  • Login.aspx constructs a redirect URL to ZavaAuthGateway (AppSettings["AuthGatewayLoginUrl"], default: http://localhost/auth/Login.aspx), appending a ReturnUrl query parameter. This is a browser-level redirect, not a server-to-server call.
  • Logout.aspx calls FormsAuthentication.SignOut() then issues a browser redirect to AppSettings["AuthGatewayLogoutUrl"] (default: http://localhost/auth/Logout.aspx).

Asynchronous / Message Queue: None detected.

Service Discovery: No service discovery mechanism. All downstream service URLs are resolved from web.config <appSettings> entries (AuthGatewayLoginUrl, AuthGatewayLogoutUrl, LedgerBaseUrl). Services are referenced by hardcoded base URLs, not logical service names.

API Gateway: None. ZavaAccountManager connects directly to ZavaLedger and ZavaAuthGateway without an intermediary gateway layer.

Client-Side Load Balancing: None detected.

Startup Dependency Chain: No startup probes or readiness checks are configured. The application starts and serves traffic immediately via xsp4. If SQL Server or ZavaLedger is unavailable at startup, failures occur at request time (not at startup). Refer to configuration-inventory.md for configuration details.

Security Posture:

  • Authentication: ASP.NET Forms Authentication is enabled. All pages require an authenticated session (cookie .ZAVAAUTH) except Login.aspx and Logout.aspx. Authentication itself is delegated to the external ZavaAuthGateway via browser redirect — ZavaAccountManager does not issue tokens or validate credentials itself.
  • Authorization: All authenticated users have access to all pages. No role-based access control ([Authorize(Roles=...)] or <allow roles="...">) is configured beyond the blanket <deny users="?" />.
  • Transport Security: No HTTPS/TLS is configured. All traffic (browser-to-app, app-to-ZavaLedger, and app-to-ZavaAuthGateway redirect) uses plain HTTP. The machineKey for Forms Auth ticket encryption is hardcoded in web.config (static key; not rotated).
  • Outbound calls to ZavaLedger are unauthenticated — no token or credential is attached to HttpWebRequest calls.

Core Business Workflows

ZavaAccountManager is a bank employee portal for managing customer accounts — enabling staff (admin and teller roles) to view customers, open and close accounts, check balances, and review transaction history as part of the Zava Bank multi-service platform.

Primary Workflows

Workflow 1: Employee Authentication

An unauthenticated employee accessing any protected page is redirected to the local Login page, which in turn redirects them to the external ZavaAuthGateway for credential verification. Upon successful authentication, ZavaAuthGateway issues a Forms Authentication cookie (.ZAVAAUTH) and redirects the employee back to their originally requested URL.

Steps:

  1. Employee requests a protected page (any page except Login.aspx and Logout.aspx).
  2. ASP.NET Forms Authentication middleware detects no valid .ZAVAAUTH cookie and redirects to ~/Login.aspx?ReturnUrl=<original URL>.
  3. Login.aspx builds the ZavaAuthGateway login URL with the ReturnUrl parameter and presents a link for the employee to continue.
  4. Employee follows the link to ZavaAuthGateway, which authenticates them and sets the .ZAVAAUTH cookie.
  5. Employee is redirected back to the original URL and is now authenticated.
  6. Session expires after 30 minutes of inactivity (sliding expiration).

Workflow 2: Customer Account Management

An authenticated staff member uses the Account Management Console (Default.aspx) to view customer data and perform account operations.

Steps:

  1. Staff member loads the Account Management Console; the system fetches the top 20 customers from the ZavaBankDb database and displays them.
  2. Staff selects a customer row; the system fetches and displays all accounts for that customer.
  3. Staff chooses one of three actions for a listed account:
    • Show Balance: Calls ZavaLedger XML API (/api/accounts/{id}/balance) to retrieve real-time balance and available balance. If ZavaLedger is unavailable, displays "Ledger balance unavailable."
    • Show Transactions: Calls ZavaLedger XML API (/api/accounts/{id}/transactions) to retrieve the transaction list. If ZavaLedger is unavailable, falls back to querying the top 25 transactions from the local Transactions table.
    • Close Account: Updates the account record in ZavaBankDb, setting Status = 'Closed' and recording CloseDate and ModifiedDate.

Workflow 3: Open New Account

An authenticated staff member opens a new bank account for a customer.

Steps:

  1. Staff member enters the Customer ID, selects an active AccountType from the dropdown, and provides an opening balance (default: 100.00).
  2. The system validates that all three fields parse successfully as integer (Customer ID), integer (AccountType ID), and decimal (balance). If validation fails, the insert is cancelled and "Invalid values." is shown.
  3. On success, the system generates an AccountNumber from the current UTC timestamp plus the Customer ID, then inserts the new account into the Accounts table with Status = 'Active' and identical Balance and AvailableBalance.
  4. The accounts grid for the selected customer is refreshed and "Account opened." is displayed.

Workflow 4: Employee Sign-Out

  1. Staff member clicks the logout link.
  2. Logout.aspx calls FormsAuthentication.SignOut() to clear the .ZAVAAUTH cookie.
  3. Employee is redirected to the ZavaAuthGateway logout URL to complete the sign-out on the identity provider side.

Cross-Service Data Flows

Balance Retrieval (ZavaLedger integration):

ZavaAccountManager calls ZavaLedger (http://zava-ledger:8080/api/accounts/{accountId}/balance) via an HTTP GET request and parses the XML response to extract //balance and //availableBalance nodes. If ZavaLedger is unreachable or returns an error, the application catches the exception and displays "Ledger balance unavailable." — no local fallback data is provided for balance queries.

Transaction History (ZavaLedger with local fallback):

ZavaAccountManager calls ZavaLedger (/api/accounts/{accountId}/transactions) and parses the XML transaction list. If ZavaLedger is unavailable, the application falls back to querying the local Transactions table in ZavaBankDb (top 25 records by date descending). This ensures staff can always view recent transaction history even when the Ledger service is down, though the fallback data may lag behind the authoritative ledger.

Authentication (ZavaAuthGateway delegation):

Authentication is fully delegated to ZavaAuthGateway. ZavaAccountManager only verifies the .ZAVAAUTH Forms Authentication cookie and reads Context.User.Identity.Name to display the signed-in employee name. It never handles passwords or credential storage.

Business Rules & Decision Logic

Validation Rules

  • Open Account input validation: Customer ID must be a valid integer, Account Type must be a selected integer value, and Opening Balance must be a valid decimal. Any parse failure cancels the insert operation.
  • Account Number generation: AccountNumber is computed as yyMMddHHmmss UTC timestamp concatenated with the zero-padded Customer ID (2 digits), ensuring uniqueness by time and customer.
  • Opening Balance equals Available Balance: When opening an account, Balance and AvailableBalance are set to the same value.

State Transitions

  • Account lifecycle: Accounts are created with Status = 'Active'. The only supported transition in this application is Active → Closed, triggered by the Close Account action. Closed accounts retain their history and are not deleted.
  • Session lifecycle: An authenticated session expires after 30 minutes of inactivity (sliding expiration). Explicit logout clears the cookie and delegates sign-out to ZavaAuthGateway.

Business Constraints

  • Customer list limit: The customer grid displays only the top 20 customers ordered by CustomerID. There is no pagination; this is a hard limit.
  • Transaction history limit (fallback): When ZavaLedger is unavailable, only the 25 most recent transactions from the local database are shown.
  • Active account types only: The Open Account form only displays AccountTypes where IsActive = 1.

Authorization

  • All pages require authentication by default (<deny users="?" />). Only Login.aspx and Logout.aspx are explicitly open to unauthenticated users.
  • Role differentiation (admin vs. teller) is managed entirely by ZavaAuthGateway; ZavaAccountManager does not enforce role-based access control beyond requiring authentication.

Error Handling

  • Ledger unavailable for balance: Exception is caught and a user-friendly message "Ledger balance unavailable." is displayed. No retry logic.
  • Ledger unavailable for transactions: Exception is caught and the application transparently falls back to querying the local Transactions table, providing continuity of service.
  • Invalid account row index: The gvAccounts_RowCommand handler validates the row index before proceeding to prevent index-out-of-bounds errors.

Transactions

  • Each database write (account insert, account close) is executed as a single SQL command with no explicit distributed transaction coordination. The application does not use a saga or compensating action pattern.

Version & Compatibility Risks

All runtime dependencies are BCL assemblies shipped with .NET Framework 4.8. While .NET Framework 4.8 is the final major release of the legacy framework and receives security fixes, it is in maintenance mode with no new feature development. ASP.NET Web Forms is not available in modern .NET (6/8/9) and has no direct migration path — porting requires rewriting the presentation layer to ASP.NET Core (Razor Pages, MVC, or Blazor). System.Data.SqlClient is the legacy SQL Server client; its modern replacement is Microsoft.Data.SqlClient (available as a NuGet package on both .NET Framework and .NET). The static machineKey values hard-coded in web.config represent a security risk and should be replaced with auto-generated or environment-supplied keys.

Notable Observations

  • No external NuGet packages: The entire dependency surface is the .NET Framework 4.8 BCL. This is unusually minimal and suggests the project avoids third-party libraries entirely, relying solely on built-in framework capabilities.
  • ASP.NET Web Forms is a dead-end technology: Web Forms is not supported on .NET 6+ and has no migration shim — any modernization to .NET 8/9 requires a full UI rewrite.
  • No ORM or query abstraction: The application uses raw System.Data.SqlClient (ADO.NET), increasing the migration surface when switching to Microsoft.Data.SqlClient or introducing an ORM such as Entity Framework Core.
  • Hard-coded machineKey in web.config: The validationKey and decryptionKey are static plain-text values, which is a known security anti-pattern; any host with access to the config file can forge authentication cookies.

No caching layer is implemented. There are no cache providers (Redis, MemoryCache, HttpRuntime.Cache, OutputCache), no [OutputCache] directives on pages, and no second-level cache configuration. Data Ownership Boundaries

ZavaAccountManager uses a shared SQL Server database (ZavaBankDB) as its sole persistence store. There is no database-per-service isolation — all tables (Customers, Accounts, AccountTypes, Transactions) reside in the same database instance and are accessed directly by this application.

Cross-service data access: Balance and transaction data have a dual source:

  • Primary: The zava-ledger external service is queried via HTTP GET with XML responses for live balance and transaction data.
  • Fallback: The local Transactions table in ZavaBankDB is queried directly when the ledger service is unreachable, indicating a partial data duplication between services.

Read/write patterns: This service performs both reads and writes on Accounts (INSERT for open, UPDATE for close) and reads on Customers, AccountTypes, and Transactions. No CQRS or event-sourcing pattern is in use.
