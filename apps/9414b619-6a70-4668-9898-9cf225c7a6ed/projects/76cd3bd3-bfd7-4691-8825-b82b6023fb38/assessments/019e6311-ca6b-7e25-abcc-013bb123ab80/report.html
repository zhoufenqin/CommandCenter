Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) accessed via raw ADO.NET with inline SQL queries. No ORM or connection pooling abstraction is present. Two external services are integrated: ZavaAuthGateway (an external identity provider, configured via AuthGatewayLoginUrl and AuthGatewayLogoutUrl) handles all user authentication and sign-out redirects, while ZavaLedger (a REST/XML service at zava-ledger:8080) exposes account balance and transaction history. The application falls back to direct database queries when the Ledger service is unavailable.

Key Architectural Decisions

  • No separation of concerns: All business logic and data access code is embedded directly in ASPX code-behind files with no distinct service or repository layer.
  • Direct ADO.NET with inline SQL: The application issues raw SQL statements using SqlConnection/SqlCommand, with no ORM (e.g., Entity Framework) and no parameterized query abstraction layer.
  • Delegated authentication: User login and logout are fully delegated to an external ZavaAuthGateway service; the application relies on the resulting Forms Authentication cookie for authorization.

DTOs & Contracts

ZavaAccountManager has no DTO or model classes. Data flows directly between the UI controls (GridView, FormView) and raw ADO.NET DataTable objects populated from SQL queries or XML parsed via XmlDocument.

  • No OpenAPI/Swagger specification is present.
  • No protobuf or GraphQL schemas are used.
  • Serialization: XML deserialization from ZavaLedger responses is handled manually using XmlDocument.SelectSingleNode / SelectNodes XPath expressions — no typed contract or schema validation.
  • Ledger response (informal contract):
    • Balance response root contains //balance and //availableBalance text nodes.
    • Transactions response contains //transaction nodes each with transactionId, amount, description, referenceNumber, transactionDate, and status child nodes.
  • There are no gateway-level aggregation DTOs or service-level domain entity classes in this codebase.

Communication Patterns

Synchronous communication:

  • Default.aspx calls ZavaLedger synchronously using System.Net.HttpWebRequest (blocking I/O on the ASP.NET thread pool). No timeout is configured on the HttpWebRequest, meaning a slow or unresponsive ZavaLedger can block threads indefinitely.
  • All database operations use SqlConnection / SqlCommand / SqlDataAdapter directly (ADO.NET, no ORM).

Asynchronous communication: None — no message queues, event buses, or async patterns are used.

Resilience patterns:

  • No circuit breaker, retry policy, or bulkhead is configured.
  • Partial fallback: If the ZavaLedger /api/accounts/{id}/transactions call fails (any exception), the application falls back to querying the local SQL Server Transactions table directly (top 25 rows by date). No fallback exists for the balance endpoint — it returns the string "Ledger balance unavailable." on failure.
  • No timeout is set on HttpWebRequest; the default .NET timeout (100 seconds) applies.

Service discovery: None — ZavaLedger is addressed by a hardcoded base URL from web.config (LedgerBaseUrl, defaulting to http://zava-ledger:8080). ZavaAuthGateway URLs are similarly config-driven (AuthGatewayLoginUrl, AuthGatewayLogoutUrl). Service names suggest Docker/Compose DNS (zava-ledger), but no Compose or Kubernetes manifest is present in this repository.

API gateway / Authentication gateway:

  • ZavaAccountManager does not act as an API gateway. Authentication is fully delegated to ZavaAuthGateway via Forms Authentication redirects. On Login.aspx, the user is presented a hyperlink to ZavaAuthGateway; there is no credential input within this application. On Logout.aspx, FormsAuthentication.SignOut() clears the local cookie, then redirects to ZavaAuthGateway logout.

Security posture:

  • Forms Authentication is configured with cookie name .ZAVAAUTH, protection="All" (encryption + validation), 30-minute sliding expiration, and SHA1/AES machine key.
  • All pages require authentication by default (<deny users="?" />); Login.aspx and Logout.aspx are explicitly opened to anonymous users.
  • No HTTPS/TLS is configured — all communication (browser ↔ app, app ↔ ZavaLedger, app ↔ ZavaAuthGateway) is over plain HTTP.
  • No authorization roles or RBAC checks are implemented within the application code; any authenticated user can access all account management functions.
  • The machineKey (validationKey and decryptionKey) is hardcoded in plain text in web.config, posing a significant key-exposure risk.

Primary Workflows

Workflow 1: Staff Authentication

When a user visits the application, ASP.NET Forms Authentication checks whether a valid .ZAVAAUTH cookie is present. If unauthenticated, the user is redirected to Login.aspx, which presents a link to the external ZavaAuthGateway login page with a ReturnUrl parameter. After the gateway authenticates the user, the browser is redirected back to the originally requested URL with a valid authentication cookie. The master page then displays the signed-in username.

Steps:

  1. User requests any protected page.
  2. Page_Load detects unauthenticated request → redirects to Login.aspx?ReturnUrl=<original>.
  3. Login.aspx builds the AuthGateway URL including ReturnUrl and displays it to the user.
  4. User follows the link to ZavaAuthGateway and authenticates.
  5. AuthGateway issues the .ZAVAAUTH forms authentication cookie and redirects back to ReturnUrl.
  6. User arrives at the protected page, now authenticated.

Logout:

  1. User clicks Logout.
  2. Logout.aspx calls FormsAuthentication.SignOut() to clear the cookie.
  3. User is redirected to the AuthGateway logout URL.

Workflow 2: Customer Selection and Account Browsing

The main dashboard (Default.aspx) allows staff to browse the most recent 20 customers and select one to view their accounts.

Steps:

  1. Page loads → retrieves top 20 customers from ZavaBankDb (Customers table) and binds them to a grid.
  2. Active account types are retrieved from AccountTypes (where IsActive=1) and bound to the "Open Account" form's dropdown.
  3. Staff clicks a customer row → gvCustomers_SelectedIndexChanged fires.
  4. Selected CustomerID is stored in ViewState for subsequent operations.
  5. Accounts for the selected customer are loaded from the Accounts table and displayed.

Workflow 3: Open New Account

Staff can open a new bank account for a customer by providing a customer ID, selecting an account type, and entering an opening balance.

Steps:

  1. Staff fills the "Open Account" form: Customer ID, Account Type (dropdown), Opening Balance.
  2. fvOpenAccount_ItemInserting validates input — all three fields must parse to valid integers/decimal; if not, the operation is cancelled with an error message.
  3. A unique account number is generated: yyMMddHHmmss + zero-padded CustomerID.
  4. A new Accounts row is inserted with Status='Active', Balance and AvailableBalance set to the opening balance, and all date fields set to the current UTC time.
  5. The accounts grid is rebound to reflect the new account.
  6. Status label confirms "Account opened."

Workflow 4: Close Account

Staff can close an active account for the selected customer.

Steps:

  1. Staff clicks the "Close Account" command button on an account row.
  2. gvAccounts_RowCommand resolves the AccountID from the grid's data keys.
  3. An UPDATE statement sets Status='Closed', CloseDate=GETDATE(), and ModifiedDate=GETDATE() for the account.
  4. The accounts grid is rebound.
  5. Status label confirms "Account closed."

Workflow 5: View Account Balance

Staff can retrieve the real-time balance for a selected account from the external Ledger service.

Steps:

  1. Staff clicks "Show Balance" on an account row.
  2. An HTTP GET request is made to LedgerBaseUrl/api/accounts/{accountId}/balance.
  3. The response is parsed as XML; //balance and //availableBalance nodes are extracted.
  4. The balance is displayed in the status label.
  5. If the Ledger service is unavailable (exception), the label shows "Ledger balance unavailable."

Workflow 6: View Transaction History

Staff can view recent transactions for an account, preferring the Ledger service with a local database fallback.

Steps:

  1. Staff clicks "Show Transactions" on an account row.
  2. An HTTP GET request is made to LedgerBaseUrl/api/accounts/{accountId}/transactions.
  3. The XML response is parsed; each <transaction> node is mapped to a row (TransactionID, Amount, Description, ReferenceNumber, TransactionDate, Status).
  4. If the Ledger service is unavailable, the application falls back to a direct SQL query: top 25 transactions from the local Transactions table ordered by TransactionDate DESC.
  5. Results are bound to the transaction history grid.

Cross-Service Data Flows

ZavaAccountManager → ZavaAuthGateway:

ZavaAccountManager does not perform credential validation itself. It delegates authentication entirely to ZavaAuthGateway via a browser redirect with a ReturnUrl. The gateway issues a Forms Authentication cookie that ZavaAccountManager trusts via shared machine key configuration (web.config).

ZavaAccountManager → ZavaLedger:

For balance and transaction data, ZavaAccountManager calls the Ledger service's XML REST API over HTTP:

  • GET /api/accounts/{id}/balance → returns XML with <balance> and <availableBalance>.
  • GET /api/accounts/{id}/transactions → returns XML with a list of <transaction> elements.

Fallback behavior: If the Ledger service is unreachable (any exception), the balance display shows a static "unavailable" message, and transaction history falls back to a direct SQL query against the local Transactions table in ZavaBankDb. This means transaction data may be incomplete or stale when the Ledger is down, but the application remains partially functional.

Business Rules & Decision Logic

Validation Rules

  • Open Account input validation: All three fields (Customer ID, Account Type, Opening Balance) must be present and parse successfully — Customer ID as integer, Account Type as integer, Opening Balance as decimal. If any field is invalid, the operation is cancelled and an error message is shown.
  • Account number generation: Account number is auto-generated as yyMMddHHmmss + zero-padded Customer ID (at least 2 digits). This ensures uniqueness based on creation timestamp.

State Transitions

  • Account lifecycle: Active → Closed. Accounts are created with Status='Active'. Closing an account sets Status='Closed' and stamps CloseDate. There is no reactivation workflow in this application.

Authorization

  • All pages require authentication: The web.config globally denies anonymous users (<deny users="?" />). Only Login.aspx and Logout.aspx allow unauthenticated access.
  • Authentication is delegated: ZavaAccountManager trusts the .ZAVAAUTH cookie issued by ZavaAuthGateway using a shared machine key. No in-app credential management occurs.
  • Session timeout: Forms authentication cookie expires after 30 minutes of inactivity (sliding expiration enabled).

Error Handling & Fallback

  • Ledger service unavailable (balance): Caught via generic exception handler; displays "Ledger balance unavailable." No retry logic.
  • Ledger service unavailable (transactions): Caught via generic exception handler; falls back to a direct SQL query against the local Transactions table, returning up to 25 most recent records by date.
  • Invalid grid row index: The gvAccounts_RowCommand handler validates the row index before acting, preventing out-of-bounds access.

Data Integrity

  • Opening balance parity: When opening an account, Balance and AvailableBalance are both set to the same opening balance value.
  • Customer-Account relationship: The application enforces foreign key integrity implicitly through SQL Server; a staff member manually inputs the Customer ID when opening an account, with no UI-level lookup validation beyond integer parsing.

Version & Compatibility Risks

The application targets .NET Framework 4.8, which is the final version of the .NET Framework and is in long-term maintenance mode (no new features). While .NET Framework 4.8 itself still receives security patches, ASP.NET Web Forms is a legacy technology with no equivalent in modern .NET (6/8/9). Migration to ASP.NET Core (Razor Pages or MVC) would require a full rewrite of the UI layer. The hardcoded machineKey values in web.config pose a critical security risk — these keys are used for Forms Authentication ticket encryption and should never be committed to source control with static values. The SQL Server connection string with embedded credentials (User Id=sa;Password=Zava123!) is another high-severity concern.

Notable Observations

  • No third-party NuGet packages: The project relies entirely on built-in .NET Framework assemblies — there is no dependency management complexity, but it also means no modern libraries (e.g., logging frameworks, ORM, DI container).
  • Hardcoded secrets in web.config: Both the machineKey (validationKey + decryptionKey) and the SQL Server sa password are stored in plain text in web.config, which is a critical security vulnerability.
  • Legacy ASP.NET Web Forms: System.Web and Web Forms have no migration path to .NET 6+; modernization requires rewriting to ASP.NET Core MVC or Razor Pages.
  • Raw ADO.NET with SQL Server: No ORM is used; database access relies on System.Data directly. The connection string targets SQL Server with TrustServerCertificate=true and a hardcoded sa account.
